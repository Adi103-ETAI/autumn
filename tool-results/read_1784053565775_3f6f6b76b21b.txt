    12→## 1. Entry point
    13→
    14→`index.html` (`out/renderer/index.html`) is a minimal shell:
    15→
    16→```html
    17→<script type="module" crossorigin src="./assets/index-C29xCgPS.js"></script>
    18→<link rel="stylesheet" crossorigin href="./assets/index-Bo6j6Brx.css">
    19→...

    22→
    23→- The single `<script type="module">` loads the **main entry bundle `assets/index-C29xCgPS.js`** (~1.5 MB). It contains the whole React app, the Zustand store, all shared component code, the Lucide icon set, analytics, voice intent parser, etc.
    24→- Mount happens at `index-C29xCgPS.js:30087`:
    25→  ```js
    26→  ReactDOM.createRoot(document.getElementById("root")).render(
    27→    <We.StrictMode><ErrorBoundary><App/></ErrorBoundary></We.StrictMode>
    28→  );
    29→  ```
    30→- `ErrorBoundary` (`index-C29xCgPS.js:30024`) catches React crashes and forwards them to the main process via `window.october.reportError(...)`. A global `unhandledrejection`/`error` listener (`index-C29xCgPS.js:30063`) also forwards crashes (a known xterm "dimensions" sync error is deliberately ignored).

    80→## 2. State management — single Zustand store
    81→
    82→There is exactly **one application store**: `useAppStore`, created with `create((set, get) => ({...}))` at `index-C29xCgPS.js:14094`, closing at `index-C29xCgPS.js:18042`. (The only other `zustand` usage in the bundle is `@xyflow/react`'s internal store inside `CanvasView-CfgQSWLU.js`.)
    83→

   228→## 5. Real-time collaboration — LiveBlocks
   229→
   230→LiveBlocks is the realtime multiplayer layer for the canvas. The client is bundled into `CanvasView-CfgQSWLU.js` (`@liveblocks/client` + `@liveblocks/react`).
   231→
   232→- **Auth bridge:** `liveblocksAuth(room)` (`CanvasView-CfgQSWLU.js:66116`) simply delegates to `window.october.liveblocks.auth(room)` (IPC `liveblocks:auth`). The room id is `canvas-${canvasId}` where `canvasId = useAppStore.getState().ensureBusCanvasId()` (a `crypto.randomUUID()` persisted in the doc as `busCanvasId`).
   233→- **Provider:** `<LiveblocksProvider authEndpoint={liveblocksAuth}>` wraps a `<RoomProvider id={"canvas-"+canvasId} initialPresence={{cursor:null,selection:[]}} initialStorage={emptyCanvasStorage()}>` (`CanvasView-CfgQSWLU.js:66123-66139`).
   234→- **Storage model** is built from the canvas doc via `LiveObject`/`LiveList`/`LiveMap` (`:57797`+). `CanvasStorageSync` (`CanvasView-CfgQSWLU.js`) mirrors the Zustand store ↔ LiveBlocks storage. Node types stored: `screen`, `chat`, `sticky`, `analytics`, `youtube`, `remotion`, `terminal`; plus `connections`, `navLinks`, `experiments` as `LiveList`s. `LOCAL_ONLY_FIELDS` are stripped before publish so per-machine/transient state (messages, busyTaskId, cwd, worktree) is never synced (`:66095-66115`).
   235→- **Presence/awareness:** `MyCursorTracker`, `SelectionTracker`, `CollaboratorTracker`, `CursorsOverlay`, `SelectionOverlay`, `AvatarsPanel` render remote cursors/selections/avatars. Cursor color from `CURSOR_PALETTE` (`CanvasView-CfgQSWLU.js:66119`).
   236→- The store's `applyRemoteCanvas` (`:14620`) is the *fallback/non-LiveBlocks* merge path (e.g. for the `bus:*` october-send flow and code-sync guests); LiveBlocks is the primary path.
   237→- Related IPC namespaces: `bus:*` (node registration, edges, october-send, cloud sync), `code-sync:*` (host/guest local↔cloud repo sync), `cloud:*` (shared canvases).

   252→## 7. xterm.js terminal rendering & data flow
   253→
   254→Terminals are **xterm.js** instances (`Terminal`, `FitAddon`, `WebLinksAddon` — `CanvasView-CfgQSWLU.js`) embedded inside `TerminalNode`. Data flows over the `terminal:*` IPC channels.
   255→
   256→Flow (terminal component, `CanvasView-CfgQSWLU.js:43020-43129`):
   257→1. On mount the component calls `window.october.terminal.start({ id, cwd, cols, rows, startCommand, agent, busCanvasId })` (`:43051`). The main process spawns a PTY and returns `{ scrollback, seq }`; scrollback is written into xterm (`:43059-43069`). `agent` is derived from the harness (`claude-code`→`"claude"`, else the harness name) unless it's a remote/gated harness (`:43047`).
   258→2. **Output:** `window.october.terminal.onData(e => …)` (`:43036`, channel `event:terminal-data`) writes `e.data` to xterm when `e.id === id`. While the node is offscreen an `IntersectionObserver` queues output (`writeOrQueue`/`hiddenQueue`, cap `HIDDEN_QUEUE_CAP`) and resyncs (re-`attach()`) on reveal (`:43073-43106`).
   259→3. **Input:** `xterm.onData(data => window.october.terminal.input(id, data))` (`:43101`, channel `terminal:input` — `send`).
   260→4. **Title:** `xterm.onTitleChange` → `updateTerminal(id,{title})` (`:43107`).
   261→5. **Exit:** `window.october.terminal.onExit(e => …)` (`:43094`, channel `event:terminal-exit`) prints `[process exited …]` and marks health.
   262→6. **Resize:** a `ResizeObserver` → `fit()` → `window.october.terminal.resize(id, cols, rows)` (channel `terminal:resize` — `send`).
   263→7. Health/attention events (`event:terminal-exit/attention/reveal`, `terminal:snapshot/kill/restart`, `foreground`, `background`) are also consumed; `busCanvasId` is passed so terminals register as multiplayer "terminal" nodes via `bus:registerNode`.
   264→
   265→`october.terminal` also exposes `whichAgents`, `scanExternalSessions`, `seedResume`, `sessionExists`, `peekSession`, `foregroundCached`, `background`/`backgroundStatus` (for keeping terminals alive after project close).
